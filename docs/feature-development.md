# 新機能開発の進め方

## 基本方針

- 新機能は1機能ずつ、小さく計画してから実装する。
- 既存の `app/` 構成、Supabase連携、Playwright E2Eを前提に進める。
- Next.js関連のAPIや構成を触る場合は、先に `node_modules/next/dist/docs/` の該当ガイドを読む。
- 余計なリファクタや既存分割の再検討はしない。

## 1機能ごとの作業テンプレート

1. 目的確認
   - 何をできるようにするか
   - 誰が使うか
   - 成功条件は何か

2. 影響範囲確認
   - UIだけか
   - Supabaseの読み書きが必要か
   - API route、認証、決済、通知、E2Eに影響するか

3. 実装方針
   - 既存コンポーネント・型・ユーティリティを優先して使う
   - 変更は対象機能に必要な範囲だけにする
   - DBや外部APIが絡む場合は失敗時表示も決める

4. 確認
   - 最小コマンドを実行する
   - 必要な場合だけE2Eを追加・更新する
   - 画面操作が絡む機能は主要導線を手動確認する

## Plan Modeを使う時

- 新機能の仕様がまだ曖昧な時
- DB、認証、決済、外部API、複数画面にまたがる時
- UI/UXの選択肢が複数ある時
- E2E追加が必要そうな時
- 実装前に「何を作るか」を固めたい時

## 通常モードでよい時

- 文言変更、軽微なUI調整
- 明確なバグ修正
- 既存機能の小さな挙動修正
- テスト追加だけ
- 影響範囲が1箇所で、仕様判断がほぼ不要な時

## 毎回の最小確認コマンド

- 基本: `npm run lint`
- UI/API/型に触った時: `npm run build`
- 主要導線やログイン後画面に影響する時: `npm run test:e2e`

`npm run test:e2e` は build も含むため、E2Eを回す時は別途 `npm run build` を重複実行しない。

```md
-

-

- なし / Supabaseに保存 / 未定

- 不要 / 必要 / 未定

成功条件:
- 

E2E確認:
- 必要そう / 不要そう / 相談したい

```

## 前提

- 当面は `app/` 配下の既存構成を維持する。
- 新機能ごとに必要最小限のファイルだけ触る。
- E2Eは「主要導線が壊れていないこと」を守る目的で、必要時だけ追加する。
